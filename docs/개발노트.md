### Phase A — KREAM 이메일 로그인 (종료)

**판정: Actions 탈락 확정. 코드 자체는 정상 작동.**

- **구현 위치**: `test/phase_a_kream_login/`
  - `kream_login.py` — 필수헤더 획득 + 이메일 로그인 API 호출

  - `KREAM_헤더_필수값_가져오기` (NUXT config → apiVersion/buildVersion/webRequestSecret/webDid)
  - `KREAM_login_이메일_request` (POST /api/auth/login)

#### 실측 이력 (2026-04-19)

| # | 환경 | IP | 결과 |
|---|------|-----|------|
| 1 | GitHub Actions | 52.173.178.211 (Azure centralus) | `/login` GET **HTTP 500, 본문 공백**, 3회 재시도 전부 실패 |
| 2 | 로컬 (가정 IP) | — | `/login` GET 200, 로그인 API 200, **access_token 수신 (491ms)** |

**결론**
- 로컬 성공 + Actions 실패 → 유일한 차이는 IP → **KREAM 이 Azure IP 대역 차단** 확정
- 500 + 본문 공백 + 재시도 전부 실패 = 일시 장애 아님, 의도적 drop 패턴
- **GitHub Actions는 실행환경에서 탈락**

#### 후속 조치

### 설계 원칙
- **한 파일 = 한 책임** (절대원칙). 변경 이유가 다르면 파일 분리.
- **세션 신선도는 session.py 한 파일이 전담** — KREAM이 정기 로그아웃시키므로 로그인+유지+헤더공급은 한 책임.
- **base/ 공통 추상화는 미루기** — KREAM 안정 후 poizon 구현 시점에 진짜 공통점 추출.

| `products.py` | CODE/SIZE → KREAM product_id 매핑 | 🔲 예정 |

### session.py 핵심 설계

- **내부 상태 `_state` (dict)** + **`_lock` (threading.Lock)**
- Public API 4개만: `configure()`, `get_headers()`, `force_refresh()`, `is_valid()`, `snapshot()`
- 선제적 재로그인 주기: **90분** (KREAM 약 2시간 만료, 여유 있게)
- 외부 모듈은 `_state` 직접 접근 금지 — get_headers() 로만 공급받음
