### 설계 원칙
- **한 파일 = 한 책임** (절대원칙). 변경 이유가 다르면 파일 분리.
- **세션 신선도는 session.py 한 파일이 전담** — KREAM이 정기 로그아웃시키므로 로그인+유지+헤더공급은 한 책임.
- **base/ 공통 추상화는 미루기** — KREAM 안정 후 poizon 구현 시점에 진짜 공통점 추출.

## 10. sell_asks.py / prices.py 구현 메모 (2026-04-19)

### sell_asks.py Public API
- `기본_배송지_ID() -> int`
- `내_판매입찰_목록(상태_필터=None) -> list[dict]`  (None/"입찰중"/"만료")
- `판매입찰_등록(product_id, option, price, *, shipping_address_id=None, is_keep_on_deferred=False, expires_in=180) -> dict`
- `판매입찰_가격수정(ask_id, 새_가격) -> dict`
- `판매입찰_삭제(ask_id) -> dict`

반환 규격: `{"성공": bool, ...}` 로 통일. 실패 시 `"사유"` 키 포함.

### prices.py Public API
- `판매입찰_시세(product_id, option) -> dict`
  - `lowest_ask_normal` (KREAM 공식 일반 판매 최저가)
  - `highest_bid` (즉시판매가)
  - `asks` (판매입찰 원시 리스트. 단, 보판 제외 시 보통 비어있을 수 있음 — 아래 주의)
- `경쟁자_최저가(product_id, option, 내_입찰가들=None) -> int|None`

### 주의: KREAM asks 엔드포인트 특성
- `GET /api/p/products/{id}/{option}/asks` 응답의 `is_immediate_delivery_item` 플래그로
  일반/보판을 구분하는데, 2026-04-19 테스트(product_id=13100, option="230") 결과
  해당 상품에선 `asks.items` 가 전부 보판이거나 비어있음. 그래도 `sales_options.lowest_normal`
  는 정상 수신되므로, **리프라이싱에 필요한 경쟁최저가는 `lowest_normal` 을 후보에 포함**하는
  방식으로 보완해 문제 없음.
- 상품별로 일반 asks 가 응답에 포함되는 경우도 있을 수 있으니, 실제 운영 상품으로 다시 검증 필요.
